# Firefighting Strategies on a 30-Vertex Planar "Map" Graph

## 1) Problem Setup

We use a fixed-size planar graph with `n = 30` vertices and run a turn-based fire containment simulation.

### State colors
- **WHITE**: untouched/unburned/unprotected
- **RED**: burning/burned
- **GREEN**: protected (blocked from burning)

### Turn dynamics
1. A strategy selects candidate vertices to protect.
2. Engine protects the first `protect_per_turn` WHITE vertices (default: `2`).
3. Fire expands: every WHITE neighbor of any RED vertex becomes RED.
4. Stop when no WHITE vertex is adjacent to RED.

### Fire front definition
Let:
`F = { v | state(v)=WHITE and exists u in N(v) with state(u)=RED }`

Only WHITE vertices in `F` are immediate one-step risk vertices.

---

## 2) Baseline Strategies

## A. `max_degree`
**Core idea:** Prioritize front vertices with largest degree (many incident edges), hoping each protection blocks more potential spread routes.

**Pseudocode:**
```text
function select_vertices(G, state):
    F <- fire_front(G, state)
    sort F by (-degree(v), v)
    return F
```

**Characteristics:**
- Deterministic and fast.
- Strong when hubs appear on the fire front.
- Can miss narrow bottlenecks if those have low degree.

## B. `min_degree_front`
**Core idea:** Prioritize low-degree front vertices, trying to seal peripheral choke tips.

**Pseudocode:**
```text
function select_vertices(G, state):
    F <- fire_front(G, state)
    sort F by (degree(v), v)
    return F
```

**Characteristics:**
- Deterministic and cheap.
- Can work in sparse/local branches.
- Often weaker on dense fronts because it ignores central connectors.

## C. `max_white_neighbors`
**Core idea:** Protect front vertices that still touch many WHITE vertices, preserving larger safe territory.

**Pseudocode:**
```text
function white_neighbors(v):
    return |{u in N(v) : state(u)=WHITE}|

function select_vertices(G, state):
    F <- fire_front(G, state)
    sort F by (-white_neighbors(v), -degree(v), v)
    return F
```

**Characteristics:**
- Looks one-step ahead into "salvageable area."
- Usually better than random.
- Still local (no global cut awareness).

## D. `random_front`
**Core idea:** Randomly choose from current fire front.

**Pseudocode:**
```text
function select_vertices(G, state):
    F <- list(fire_front(G, state))
    shuffle(F)
    return F
```

**Characteristics:**
- Baseline for comparison.
- Non-deterministic unless RNG seed fixed.
- Useful for robustness experiments across many runs.

## 3) Creative Min-Cut-Inspired Strategies

## E. `min_cut_edge_front`
**Core idea:** Build an augmented graph with super-source connected to RED vertices and super-sink connected to WHITE vertices; compute a minimum **edge cut** between them. Protect WHITE endpoints on that cut, prioritizing frontier endpoints.

**Interpretation:** The cut edges approximate a minimal "firewall" that separates burning territory from safe territory.

**Pseudocode:**
```text
function select_vertices(G, state):
    F <- fire_front(G, state)
    R <- {v : state(v)=RED}
    W <- {v : state(v)=WHITE}
    if F empty or R empty or W empty: return F

    G' <- copy of G
    add source s, sink t
    for r in R: add edge (s, r)
    for w in W: add edge (w, t)

    C_edges <- minimum_edge_cut(G', s, t)

    score[x] <- number of times WHITE vertex x appears in C_edges endpoints
    preferred <- vertices sorted by (-score[x], -degree(x), x)

    return (preferred intersect F) + (F \ preferred)
```

**Characteristics:**
- Uses a **global separator** signal, not just local degree.
- Edge cut can identify strategic barriers missed by greedy local rules.
- Approximate for this game because action protects vertices, not edges.

## F. `min_cut_vertex_front`
**Core idea:** Protect vertices from a minimum **vertex cut** separating RED set from distant WHITE "target territory."

**How target territory is chosen:** pick farthest WHITE vertices from the RED set (multi-source shortest path distances), then compute a node cut to those targets.

**Pseudocode:**
```text
function select_vertices(G, state):
    F <- fire_front(G, state)
    R <- {v : state(v)=RED}
    W <- {v : state(v)=WHITE}
    if F empty or R empty or W empty: return F

    dist <- multi_source_shortest_path_lengths(G, sources=R)
    T <- top-k vertices in W by farthest dist (k=5)

    G' <- copy of G
    add source s, sink t
    for r in R: add edge (s, r)
    for z in T: add edge (z, t)

    C_nodes <- minimum_node_cut(G', s, t)
    P <- WHITE vertices in C_nodes
    preferred <- sort P by (-degree(v), v)

    return (preferred intersect F) + (F \ preferred)
```

**Characteristics:**
- Directly targets **separator vertices**, conceptually aligned with vertex-protection rules.
- More strategic/global than purely local heuristics.
- Depends on chosen target set (`k`, distance metric), so tuning matters.

---

## 4) Why Min-Cut Strategies Are "Creative" Here

- Classic greedy methods score each front vertex independently.
- Min-cut methods reason about **global connectivity**: "what small barrier separates fire from remaining territory?"
- They can discover non-obvious protective positions even when those vertices are not top degree.

---

## 5) Complexity Notes (per turn, high level)

- `max_degree`, `min_degree_front`, `max_white_neighbors`, `random_front`: mostly sorting on `|F|`.
- `min_cut_edge_front`: one min-edge-cut on augmented graph (heavier).
- `min_cut_vertex_front`: shortest paths + min-node-cut on augmented graph (heavier).

Tradeoff: higher runtime for potentially better strategic containment.
